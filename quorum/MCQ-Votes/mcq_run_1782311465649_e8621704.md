## Question

For a small Python tabular ML template, which pragmatic architecture change best improves extensibility/readability so new capabilities can be added with minimal edits? Current pattern: estimators use a registry; data loaders use if/elif branches; preprocessing/validation/metrics are hardcoded functions/dataclass fields; feature engineering has a programmatic FeaturePipeline but is not config-wired.

## Options

- **[registries]** Small explicit registries — Use lightweight factory registries for loaders, validation rules, metrics, and selected preprocessing hooks, matching the existing estimator registry.
- **[branches]** Keep branches — Keep explicit if/elif branches and dataclass fields; improve docs/tests only.
- **[plugin_system]** Plugin system — Introduce generic plugin discovery for all extension points.
- **[config_model]** Config model first — Prioritize stronger typed config parsing/validation before changing extension-point architecture.

## Criteria

- **[future_edits]** Future edits — Reduces number of files/functions changed for new capabilities. (weight: 3)
- **[readability]** Readability — Keeps the template understandable for a small personal project. (weight: 3)
- **[scope]** Scope — Can be implemented incrementally without a broad rewrite. (weight: 2)
- **[tests]** Testability — Makes extension behavior easy to test independently. (weight: 2)
