## Question

For this small Python tabular ML template, which pragmatic change would most improve 'add a new capability with the fewest code changes' while preserving readability? Context: estimators already use ESTIMATOR_REGISTRY; data sources use if/elif in load_dataframe_from_config; preprocessing/imputers use PreprocessingConfig plus branches; validation uses DataSchema fields plus one long validate_dataframe function and ValidationIssue.__str__; FeaturePipeline exists but is only programmatic, not TOML-wired; metrics are hardcoded functions. Choose the best first architectural move.

## Options

- **[registries]** Small explicit registries — Add lightweight registries/factory maps for data loaders, imputers/preprocessing builders, validation rule functions, metrics, and optional feature factories; keep config parsing explicit.
- **[leave_explicit]** Keep explicit branches — Keep current if/elif and dataclass-field approach because the repo is small; improve docs and comments only.
- **[plugin_framework]** Plugin framework — Build a generic plugin system/discovery layer for loaders, features, validators, preprocessors, metrics, and estimators.
- **[config_schema_model]** Typed config model first — Introduce a stronger config modeling/validation library and focus on reducing project_config parsing boilerplate before adding registries.

- **[fewest_changes]** Fewest changes for future features — How much the option reduces future code edits for new capabilities. (weight: 3)
- **[readability]** Readability — How understandable it remains for a small personal ML template. (weight: 3)
- **[scope]** Implementation scope — How small and pragmatic the initial refactor is. (weight: 2)
- **[testability]** Testability — How easy it is to test each extension independently. (weight: 2)
